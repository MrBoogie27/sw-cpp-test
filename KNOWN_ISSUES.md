# **Известные недостатки и сомнения по решению**
1. Для расширяемости хотелось бы завести EventBus с подпиской на события. Для этого необходимо поменять код в IO – сделать интерфейсы для событий и обработчиков этих событий.
2. Настройка действий происходит только на этапе создания, приоритет по порядку добавления – для простоты.
3. Можно вызывать `issueMarch` для юнита, который не может двигаться. Не критично, но может вводить в заблуждение.
4. Можно создать интерфейсы для карты и симуляции. Для расширения и тестирования. Пока не требуется.
5. Карту/мир хочется абстрагировать на случай появления 3D мира, или мультимиров. Симуляция при этом останется одна.
6. Необходим более сложный поиск пути передвижения. Можно BFS, но тоже могут быть проблемы.
7. Неоптимальный алгоритм выбора врага. Можно смотреть сразу ближайших. Сейчас выбор детерминированный (не случайный как по ТЗ).
8. Юнит может быть убит, но после этого двигаться – странно, но по ТЗ так.
